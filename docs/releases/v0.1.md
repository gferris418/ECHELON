# Project ECHELON Documentation Snapshot — v0.1

> **Status:** Historical baseline. This document records the original Foundation Bible and must not be silently rewritten. Later decisions are recorded in v0.2 and v0.3.

## 1. Mission

Project ECHELON is a modular, self-hosted intelligence platform where a human operator can command software, AI specialists, knowledge, and infrastructure from one visual interface.

ECHELON is not defined by any single agent runtime, chat interface, model provider, or automation tool. Its permanent value is the platform kernel, documented standards, capability model, and human approval rules.

## 2. Platform constitution

1. **Core first.** The kernel is stable; installed capabilities are replaceable.
2. **One capability, one best tool.** Do not add two tools that solve the same primary problem.
3. **Human authority.** AI may propose, plan, execute approved actions, and verify results. It must not silently perform high-impact actions.
4. **Open interfaces.** Capabilities should expose APIs, health checks, events, and standard integration points.
5. **Event-driven communication.** Components should publish and consume events rather than create hidden direct dependencies.
6. **Workspace isolation.** A workspace may isolate knowledge, credentials, storage, models, capability access, and permissions.
7. **Observable operations.** Every meaningful action produces a task record, event, log, and audit entry.
8. **Documentation before implementation.** No code or configuration is considered approved until its design, phase, and acceptance criteria are recorded.

## 3. Terminology

| Standard term | Meaning |
|---|---|
| Mission Control | Main operational interface |
| Kernel | Headless permanent core services |
| ORACLE | Voice/text assistant and non-technical operator guide |
| Capability | Replaceable installed product, service, adapter, or module |
| Foundry | Capability catalogue and installation surface |
| Specialist | An AI or automated worker with a defined role |
| Mission | A planned, observable workflow |
| Objective | One unit of work within a mission |
| Signal | Notification or event requiring awareness |
| Timeline | Human-readable history of actions and state changes |
| Workspace | Isolated operational context |
| Knowledge Core | Governed documents, memory, and retrieved context |

## 4. System architecture

```text
User (voice or text)
        |
Mission Control + ORACLE
        |
ECHELON Kernel
        |
+-- Identity and authorization
+-- Capability registry and lifecycle manager
+-- Task engine and scheduler
+-- Event bus
+-- Secrets interface
+-- Configuration service
+-- Storage abstraction
+-- Audit trail
+-- Health/monitoring adapter
+-- Gateway abstraction
        |
Installed capabilities and external services
```

The kernel must not directly depend on Hermes, LobeHub, Agent Zero, OpenHands, n8n, or any future tool. Those systems are candidates for capability adapters.

## 5. Mission Control specification

Mission Control is the primary visual control surface. It is not an ordinary administrative dashboard.

Initial surfaces:

- System overview: service health, active tasks, alerts, capacity.
- Specialists: registered AI workers and their status.
- Capabilities: installed services, versions, dependencies, permissions, health, logs, lifecycle actions.
- Missions: queued, running, paused, failed, completed, and awaiting approval workflows.
- Timeline: unified activity and audit feed.
- Settings: workspace, access, gateway, storage, backup, and presentation settings.

Design rule: use spatial/3D visualization only when it improves comprehension, such as a dependency graph, execution flow, or knowledge relationship map. Routine controls must remain fast and readable in conventional 2D UI.

## 6. ORACLE specification

ORACLE is a kernel-level assistant, not a separate chatbot product.

ORACLE accepts voice and typed requests. It translates intent into an explicit plan, shows dependencies and risks, asks for approval where required, invokes the kernel task engine, verifies outcomes, and explains results in plain language.

Required operating sequence:

```text
Understand -> retrieve platform context -> propose plan -> visualize -> request approval -> execute -> verify -> record -> report
```

ORACLE must explain technical terms for non-technical operators. It must never claim an action succeeded without a verifiable completion signal.

## 7. Initial infrastructure choices

| Need | Baseline selection | Reason |
|---|---|---|
| Relational data | PostgreSQL | Durable configuration, users, tasks, audit records |
| Queue/cache/session state | Redis | Fast task state, cache, transient coordination |
| Object storage | MinIO | S3-compatible files, uploads, backups |
| Routing/TLS | Traefik | Service routing and discovery |
| Identity | Authentik | SSO, user roles, access policies |
| Model gateway | 9router | Existing OpenAI-compatible gateway strategy |
| Agent runtime | Hermes Agent | Autonomous execution candidate |
| AI workspace | LobeHub | Candidate conversational workspace |
| Automation | n8n | Workflow automation candidate |
| Knowledge/RAG | RAGFlow | Document retrieval candidate |
| Internal application builder | ToolJet | Future CRUD/forms/internal tools candidate |
| Engineering specialist | OpenHands | Future coding capability candidate |
| Observability | Grafana + Prometheus + Loki | Metrics, dashboards, logs |

These are decisions for the baseline, not permanent kernel dependencies.

## 8. Capability manifest standard

Every capability must supply a machine-readable manifest containing at least:

```yaml
id: example-capability
name: Example Capability
version: 0.1.0
type: workspace|agent|automation|knowledge|integration|observability
entrypoint: http://service:port
health:
  path: /health
permissions: []
dependencies: []
settings_schema: {}
events:
  publishes: []
  subscribes: []
ui:
  routes: []
update:
  strategy: managed|manual
  rollback: supported|not-supported
```

The Capability Manager may install, configure, start, stop, update, monitor, and remove capabilities only through documented lifecycle actions.

## 9. Documentation-first development protocol

Before any implementation work:

1. Read the Bible, current architecture, current phase, relevant ADRs, and existing module specifications.
2. Identify what is known, unknown, assumed, and externally verified.
3. Produce a plan, risks, dependencies, security impact, rollback plan, and acceptance criteria.
4. Obtain approval for scope and high-impact actions.
5. Implement on a branch.
6. Run tests and verify observable behaviour.
7. Update documentation, API contracts, manifests, and ADRs in the same change.
8. Commit and publish the resulting decision trail.

## 10. Phase model

- **Phase -1:** Bible, standards, ADRs, repository structure, GitBook publishing model.
- **Phase 0:** Kernel skeleton, Mission Control shell, ORACLE shell, PostgreSQL, Redis, MinIO, Traefik.
- **Phase 1:** Identity, 9router connection, initial specialist/runtime capability, initial AI workspace.
- **Phase 2:** Automation, knowledge/RAG, monitoring and logs.
- **Phase 3:** Low-code internal modules and engineering specialist.
- **Phase 4:** Capability Foundry and workspace manager.
- **Phase 5:** Mobile/PWA controls and remote approval flows.

## 11. Acceptance criteria for Phase -1

Phase -1 is complete only when:

- The Bible exists in GitHub.
- GitHub is declared the authoritative source.
- GitBook is declared a downstream publication layer.
- The glossary and terminology are stable enough for implementation.
- Every planned external component is categorized as kernel, capability, research, or rejected.
- Architecture decisions are stored as ADRs.
- AI contributors have an explicit anti-drift protocol.
