# Project ECHELON Documentation Snapshot — v0.3

> **Status:** Current documentation baseline until superseded by an approved later release.

## 1. Purpose of v0.3

v0.3 adds the **Intelligence Layer**: a governed way to collect events and context from approved sources, assemble only relevant evidence for a request, and make ORACLE more accurate without giving it unrestricted surveillance, hidden memory, or uncontrolled execution.

The Intelligence Layer extends the v0.1 kernel and v0.2 governance model. It does not change the core rule that the user remains in command for critical actions.

## 2. New permanent architecture components

### 2.1 Intelligence Bus

The Intelligence Bus is a kernel-level event contract and routing layer. It receives structured events from approved capabilities and sends them to authorized subscribers.

Potential publishers:

- Kernel task engine.
- Capability lifecycle manager.
- ORACLE.
- Hermes or other approved specialist runtimes.
- n8n automation.
- RAG/knowledge capability.
- Browser specialist.
- Screen-intelligence capability.
- Monitoring stack.
- User interface.

Potential subscribers:

- Timeline.
- Signal/notification service.
- Context Engine.
- Mission Control live views.
- Audit system.
- Approved automation or specialist capability.

Required event envelope:

```yaml
event_id: uuid
event_type: capability.health.changed
occurred_at: ISO-8601 UTC
workspace_id: optional-id
source:
  capability_id: example
  instance_id: example-01
classification: operational|sensitive|restricted
payload: {}
trace:
  task_id: optional-id
  mission_id: optional-id
  correlation_id: required-id
```

No capability may rely on undocumented event payloads. Every event type must have a schema, publisher, subscriber list, retention rule, and classification.

### 2.2 Context Engine

The Context Engine prepares evidence for ORACLE or an approved specialist. It is not a free-form memory dump.

Its responsibilities:

1. Read the active request and workspace boundary.
2. Retrieve approved, relevant sources in priority order.
3. Rank and filter evidence according to permissions, recency, relevance, classification, and token budget.
4. Record provenance: what was retrieved, why it was included, and what was excluded.
5. Construct a bounded context package for the model.
6. Preserve uncertainty instead of filling gaps with guessed facts.

Required retrieval precedence:

```text
Current task state
-> active workspace policy and permissions
-> approved Bible / ADR / specifications
-> current repository and deployed configuration
-> knowledge base and cited documents
-> recent authorized timeline/events
-> approved external sources
```

The Context Engine must provide citations or source references in operator-facing answers where possible.

### 2.3 Screen Intelligence

Screen Intelligence is an optional, opt-in capability inspired by local context-capture systems. It may help an operator find information from recently viewed work, but it is not a silent surveillance service.

Rules:

- Off by default.
- Per-workspace and per-source consent.
- Clear indicator when capture is enabled.
- Configurable source exclusions and redaction rules.
- User may pause, inspect, export, or delete collected context.
- No sensitive screen, password, banking, health, or private communications capture without explicitly configured policy and approval.
- Screen-derived data is classified and retained under a published policy.

Initial permitted function: searchable local activity timeline for user-approved workspaces.

### 2.4 Browser Specialist

The Browser Specialist is an optional capability for user-approved browser navigation, research, and repetitive web workflows.

Guardrails:

- Visible browser/session state where technically possible.
- Domain allow/deny policy.
- User approval before login, data submission, purchases, file uploads, external messages, or irreversible actions.
- Captured results must include source URLs and timestamps where available.
- Browser automation must not bypass access controls or security controls.

### 2.5 API Laboratory

The API Laboratory is an internal capability based on API-client/testing patterns such as Bruno.

Use cases:

- Test ECHELON APIs and capability adapters.
- Keep versioned request collections near code.
- Validate authentication, schemas, error handling, and performance.
- Produce safe reproducible integration tests.

Secrets must be referenced from the secret service, never committed into collections or documentation.

## 3. Capability evaluation register

All external repositories and self-hosted applications must be evaluated before installation. A candidate has one of five statuses:

| Status      | Meaning                                                                       |
| ----------- | ----------------------------------------------------------------------------- |
| Approved    | Unique capability with an accepted integration plan                           |
| Conditional | Useful but requires a defined prerequisite or limitation                      |
| Research    | Valuable for patterns or future evaluation; not installed                     |
| Rejected    | Overlaps, lacks required security/maintenance, or conflicts with architecture |
| Deferred    | Potentially useful later but not justified in the active phase                |

### Current approved capability directions

| Capability                     | Intended function                            | Basis / candidate                                                         |
| ------------------------------ | -------------------------------------------- | ------------------------------------------------------------------------- |
| Intelligence Bus               | Structured events and live operational state | Custom kernel component                                                   |
| Context Engine                 | Evidence-based context assembly              | Custom kernel component; research patterns from context projects          |
| Screen Intelligence            | Opt-in local timeline/context                | Screenpipe concepts; adapter only after policy review                     |
| Browser Specialist             | Approved web operations                      | NanoBrowser or equivalent adapter after review                            |
| API Laboratory                 | Versioned API testing                        | Bruno or equivalent                                                       |
| Hardware awareness             | System capacity and health data              | Platform-specific collector; LibreHardwareMonitor patterns where relevant |
| Voice transcription            | Speech-to-text                               | Whisper-compatible capability                                             |
| Enterprise files/collaboration | Future document capability                   | Nextcloud candidate; not required for Phase 0                             |

## 4. Repository evaluation policy

Before an external repository becomes a capability candidate, document:

1. Exact problem it solves.
2. Existing ECHELON capability that might overlap.
3. Repository activity, license, release cadence, issue health, and maintainer ownership.
4. Deployment model and system requirements.
5. Authentication, secrets, data flow, network exposure, and audit implications.
6. APIs, MCP/event support, health endpoint, and machine integration path.
7. Data portability, backup, update, rollback, and removal plan.
8. Workspace isolation implications.
9. Test plan and kill switch.
10. Decision: approved, conditional, research, rejected, or deferred.

No repository is added because it has an attractive interface or a long feature list.

## 5. Updated phase roadmap

### Phase -1 — Constitution and engineering controls

Deliverables:

- Bible snapshots.
- Machine-readable `.echelon/` context files.
- ADR log.
- Capability register.
- GitHub/GitBook publishing rules.
- Documentation integrity checklist.

Acceptance: a new human or AI contributor can determine what ECHELON is, what phase is active, which decisions are approved, and what must be verified before changing the platform.

### Phase 0 — Kernel and operator foundation

Deliverables:

- Kernel service boundaries.
- PostgreSQL, Redis, MinIO, and Traefik baseline.
- Mission Control shell.
- ORACLE text mode, then voice proof-of-concept.
- Task and Timeline records.
- Identity integration plan.
- Basic capability registry and health model.

Acceptance: a user can see health, describe a simple request, review a plan, approve it, run a safe demonstrator task, and see verified status in the Timeline.

### Phase 1 — Controlled AI execution

Deliverables:

- Authentik or approved identity capability.
- 9router gateway connector.
- One approved agent/runtime capability.
- One approved user-facing workspace capability.
- Secret reference model.
- Model usage, error, and cost telemetry.

Acceptance: ORACLE can safely direct a single specialist through the gateway while retaining user approval and a complete audit trail.

### Phase 2 — Intelligence, knowledge, automation, and observability

Deliverables:

- Intelligence Bus schemas and first publishers/subscribers.
- Context Engine with provenance.
- RAG/knowledge capability.
- n8n or approved automation capability.
- Grafana/Prometheus/Loki or approved observability stack.
- Optional voice/STT capability.
- Optional Screen Intelligence proof-of-concept with consent controls.

Acceptance: a request can be answered or executed using cited approved context, and the operator can trace why the system acted.

### Phase 3 — Builder capabilities

Deliverables:

- ToolJet or approved internal-app capability.
- OpenHands or approved engineering capability.
- API Laboratory.
- Browser Specialist proof-of-concept.

Acceptance: ECHELON can build/test a small internal module through documented approvals and observe the end-to-end execution.

### Phase 4 — Foundry and workspaces

Deliverables:

- Capability manifest validation.
- Dependency and permission checks.
- Install/update/rollback/remove flows.
- Workspace creation and isolation.
- Capability catalogue with evidence-based evaluation record.

Acceptance: an approved capability can be installed as a repeatable mission without modifying the kernel.

### Phase 5 — Mobile, remote, and edge operations

Deliverables:

- PWA/mobile operator view.
- Voice commands and approval flows.
- Secure remote access model.
- Edge/device integration policy.

Acceptance: an authorized operator can review, approve, and monitor missions securely from a mobile device.

## 6. ADR additions

### ADR-007: Intelligence Bus

**Decision:** Add a kernel-level structured event contract.

**Reason:** The platform needs a common nervous system without direct point-to-point coupling between every capability.

**Consequence:** Event schemas, classification, retention, and access must be governed as first-class architecture assets.

### ADR-008: Context Engine

**Decision:** Add a kernel-level evidence and context assembly service.

**Reason:** ORACLE and specialists must be grounded in the Bible, current state, and authorized evidence rather than relying on model memory.

**Consequence:** Context retrieval must preserve provenance and obey workspace and classification policy.

### ADR-009: Repository evaluation policy

**Decision:** Every external project is evaluated against unique value, overlap, maintenance, security, lifecycle, and removal criteria.

**Reason:** Prevent capability sprawl and architecture drift.

**Consequence:** A repository may be useful as research without becoming an installed dependency.

## 7. AI contributor rules

Before proposing or modifying a feature, the AI must:

1. State the active release and phase.
2. Identify relevant Bible sections and ADRs.
3. Search existing repository files before creating new interfaces or modules.
4. Identify overlap with existing capabilities.
5. Separate fact, assumption, inference, and proposal.
6. Produce test and rollback plans.
7. Refuse to claim runtime results that it has not verified.
8. Update the documentation and machine-readable context in the same change as implementation.

## 8. Current baseline statement

As of v0.3, ECHELON is still in **Phase -1 / documentation and architecture control**. It is not yet a deployed product. The next approved work is to convert these documents into a reviewed, machine-readable current specification, then begin Phase 0 with a small verifiable kernel prototype.
