# Project ECHELON Documentation Snapshot — v0.2

> **Status:** Historical governance and interaction update. v0.2 extends v0.1; it does not replace the v0.1 record.

## 1. Change summary

v0.2 formalized ECHELON as a conversational operating environment. The central design change is that non-technical users should be able to speak or write what they want, inspect the proposed visual workflow, approve it, and receive verified results without learning containers, ports, environment variables, or provider-specific configuration.

## 2. ORACLE is a kernel service

ORACLE is confirmed as a permanent kernel service because it is needed before optional capabilities are installed. It guides first-time setup, translates technical state into plain language, and enforces approval boundaries.

### ORACLE responsibilities

- Accept voice and text requests.
- Explain the current state of the system in plain language.
- Retrieve the Bible, ADRs, phase documentation, capability manifests, and workspace context before making technical claims.
- Build an explicit execution plan.
- Display a visual workflow showing prerequisites, risks, approval points, execution stages, and verification checks.
- Execute only through the Kernel task engine after required approval.
- Verify outcomes using health checks, task results, or other defined evidence.
- Record the plan, approval, execution events, verification result, and final report in the Timeline.

### Required confirmation actions

ORACLE must require explicit confirmation before:

- Installing an untrusted or unreviewed repository.
- Storing or rotating secrets.
- Exposing a service externally.
- Changing administrator identity or access controls.
- Deleting or overwriting data.
- Sending messages or submitting forms externally.
- Spending money or using a billable external resource beyond a configured limit.
- Running a security-sensitive or destructive operation.

## 3. Voice and visual interaction model

The user experience follows this fixed model:

```text
Voice/Text intent
      -> ORACLE interpretation
      -> Context retrieval
      -> Visual plan
      -> User approval when needed
      -> Kernel task execution
      -> Evidence-based verification
      -> Timeline and human-readable result
```

Voice should be treated as an interaction modality, not the only source of truth. Every spoken request is transcribed, displayed, and editable before high-impact execution. Every spoken result is also available in readable text and timeline evidence.

## 4. Mission Control design direction

Mission Control should feel alive, spatial, and real-time without becoming decorative or difficult to use.

### Always visible

- ORACLE conversation/voice panel.
- Active objective and current execution stage.
- Pending approvals.
- System health summary.
- Timeline updates.

### Visual layers

- **Operational layer:** cards, tables, forms, alerts, logs, and approvals for precise work.
- **Spatial layer:** dependency graph, capability map, workflow animation, specialist activity map, and knowledge graph when relationships matter.
- **Detail layer:** drill-down into exact task input, task output, logs, API responses, or configuration.

The interface must never hide the conventional controls required for accessibility or efficiency.

## 5. GitHub-first, GitBook-second governance

GitHub is the authoritative engineering memory. GitBook is a readable publication surface.

### Rules

1. A change begins in GitHub documentation or source-controlled specifications.
2. GitBook may publish or render the approved GitHub content.
3. A direct GitBook edit is temporary only; it must be reflected in GitHub before it can be treated as authoritative.
4. If GitBook and GitHub conflict, GitHub wins.
5. Every GitBook page should be traceable to a GitHub path and commit or release version.

## 6. Anti-drift / anti-hallucination protocol

An AI contributor must not work from memory alone.

Before acting, it must retrieve:

- Current release baseline.
- Relevant Bible sections.
- Applicable ADRs.
- Active phase and acceptance criteria.
- Existing source files and interfaces.
- Capability manifest and current health/configuration state, when relevant.

The AI must separate:

- **Fact:** directly supported by repository content, runtime evidence, or approved external source.
- **Assumption:** an unverified condition required to proceed.
- **Inference:** a conclusion drawn from stated facts.
- **Proposal:** a suggested future change.

If a fact is unavailable, the AI must say so rather than inventing it.

## 7. One capability = one best tool policy

ECHELON does not accumulate overlapping products by default.

A candidate can be added only when it:

1. Adds a distinct capability not already supplied by an approved component.
2. Has an active maintenance and security posture appropriate to its role.
3. Supports self-hosting or an explicitly approved hosting model.
4. Has a documented API, event model, or adapter path.
5. Can be installed, updated, monitored, and removed without changing the kernel.
6. Has a clear data, secret, and permission model.
7. Has a documented rollback or recovery path.

## 8. Capability research records

### OpenClaw Nerve

**Decision:** Research/reference candidate; do not add to the core stack yet.

Nerve offers a rich agent cockpit with voice, fleet control, workspace files, session trees, crons, kanban, cost/context views, and live output. It overlaps with Mission Control, ORACLE, and an AI workspace. Re-evaluate only if OpenClaw is adopted as an approved agent runtime or if a comparison proves that it replaces rather than duplicates an existing UI capability.

### Hermes Studio

**Decision:** Under evaluation.

Do not admit based on the repository name or assumed features. Evaluate the actual deployment model, API surface, security, maintenance, unique capability, and overlap with the chosen Hermes runtime and workspace approach.

### Hermes dashboard/tutorial references

**Decision:** Reference architecture only.

Adopt useful interaction patterns—fleet visibility, voice operation, workspace awareness, scheduled work, task boards, and live status—without copying platform-specific assumptions or fixed agent structures.

## 9. ADR additions

### ADR-004: One capability, one best tool

**Decision:** Reject overlapping products unless the new product adds an essential capability not covered by an approved selection.

**Consequence:** The stack remains smaller and easier to secure, document, and operate.

### ADR-005: ORACLE belongs in the kernel

**Decision:** ORACLE is a permanent kernel service.

**Consequence:** Every user can configure and operate ECHELON before a specialized agent or workspace capability is installed.

### ADR-006: GitHub first; GitBook second

**Decision:** GitHub is the authoritative repository. GitBook is a documentation publication layer.

**Consequence:** Documentation decisions remain versioned, reviewable, and available to AI contributors.

## 10. Phase roadmap update

- **Phase -1:** Documentation, ADRs, machine-readable context, governance gates.
- **Phase 0:** Headless kernel and initial Mission Control/ORACLE experience.
- **Phase 1:** Authentication, model gateway, one approved runtime capability, one approved workspace capability.
- **Phase 2:** Intelligence services, knowledge, automation, observability.
- **Phase 3:** Internal application and engineering capabilities.
- **Phase 4:** Foundry, capability SDK, workspace lifecycle.
- **Phase 5:** Mobile/PWA, remote approvals, edge support.

## 11. Machine-readable project context

The repository should maintain `.echelon/` files that are readable by software and AI agents. At minimum:

```text
.echelon/platform.yaml
.echelon/phases.yaml
.echelon/capabilities.yaml
.echelon/ai-rules.yaml
```

Those files summarize identity, architecture rules, active phase, capability decisions, and AI operating constraints. They supplement Markdown; they do not replace detailed documentation or code review.
